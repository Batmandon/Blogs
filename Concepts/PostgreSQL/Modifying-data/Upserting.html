The Problem: Managing Unknown Record States

When handling user data (for example preferences), you may need to insert a record if it doesn't exist, or update it if it does. A common but inefficient two-step approach is:

  1. Select: Query to check whether the record exists.
  2. Logic: If found, run UPDATE; otherwise run INSERT.

Why this is problematic

  • Performance: Requires two separate network round trips between the app and database.
  • Race conditions: A timing gap between read and write can cause data inconsistencies under concurrency.

The Solution: The Upsert

An Upsert (update + insert) combines both paths into one atomic query, preventing races and reducing load. In PostgreSQL this is done with INSERT ... ON CONFLICT.

Implementation Strategies (using ON CONFLICT)

  • Overwrite with new data: On unique key conflict, update specific columns using the excluded pseudo-table (which holds the attempted insert values).
  • Do nothing: Use DO NOTHING if you only want to ensure a record exists without changing existing data.
  • Calculate/Increment values: Reference existing table values to compute a new value instead of using excluded.

Examples

-- Overwrite with new data
INSERT INTO user_prefs (user_id, theme)
VALUES (123, 'dark')
ON CONFLICT (user_id) DO UPDATE
SET theme = excluded.theme;

-- Do nothing when a row already exists
INSERT INTO user_prefs (user_id, theme)
VALUES (123, 'default')
ON CONFLICT (user_id) DO NOTHING;

-- Increment an analytics counter
INSERT INTO view_counters (item_id, views)
VALUES (42, 1)
ON CONFLICT (item_id) DO UPDATE
SET views = view_counters.views + 1;
