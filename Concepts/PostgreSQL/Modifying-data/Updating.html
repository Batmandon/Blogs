Summary

The Danger of Unfiltered Updates — Running an UPDATE without a WHERE clause will modify every row in the target table. Writing the update expression before verifying the filter creates a "dangling update" risk that can accidentally overwrite large amounts of data (for example, setting every user's email to the same value).

The Golden Rule: Select, Then Update

Adopt a consistent workflow to avoid accidental mass updates:

  1. Write a SELECT statement: Compose a SELECT using the exact WHERE clause you intend to use for the update to preview affected rows.
  2. Verify results: Run the SELECT and confirm the returned rows match expectations. If a query intended for one row returns many, fix the logic before proceeding.
  3. Convert to UPDATE: After verification, replace the SELECT with an UPDATE, keeping the vetted WHERE clause intact.

Additional Tips

  • Complex logic: This practice is especially important with complex conditions (dates, intervals, multiple booleans).
  • RETURNING clause: PostgreSQL supports RETURNING * on UPDATE to see modified rows immediately.
  • Consistency: Apply this workflow even for simple updates — complacency causes mistakes.

Example

-- verify
SELECT * FROM users WHERE id = 123;

-- update after verification
UPDATE users
SET email = 'new@example.com'
WHERE id = 123
RETURNING *;
