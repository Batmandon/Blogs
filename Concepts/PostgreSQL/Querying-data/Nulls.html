Handling NULL Values in PostgreSQL

This page explains the core concepts of NULL in PostgreSQL, why it behaves differently from normal values, and how to work with it safely in queries.

Core Concept: NULL is Unknown

NULL is not a value, and it is not equivalent to false, an empty string, or zero. It represents an unknown or missing value, like a "black box." This distinction is crucial for correct SQL logic.

Three-Valued Logic

SQL uses a third logical state: Unknown. This changes how comparisons behave when NULL is involved.

  • Standard logic: 5 = 5 is True; 5 = 10 is False.
  • NULL logic: 5 = NULL evaluates to NULL (Unknown).
  • Null equality: NULL = NULL also evaluates to NULL, because two unknowns cannot be assumed equal.

Filtering NULL Values

Queries only return rows when the WHERE clause evaluates to strictly True. Expressions that resolve to NULL are treated as Unknown and do not pass the filter.

  • Incorrect: WHERE phone = NULL returns zero rows because the expression evaluates to NULL, not True.
  • Correct: Use IS NULL or IS NOT NULL to filter rows with unknown values.

Handling NULLs with COALESCE

The COALESCE function returns the first non-NULL value from its arguments, making it useful for fallbacks and safe display values.

  • Functionality: It accepts multiple arguments and returns the first non-null value.
  • Usage: Use it to provide defaults, for example COALESCE(phone, 'No phone given') or COALESCE(name, username).

Nulls in Arithmetic

Any arithmetic operation involving NULL results in NULL. To prevent broken calculations, wrap nullable values with COALESCE and provide a default.

  • Example: 10 + NULL yields NULL.
  • Safe replacement: COALESCE(column, 0).
