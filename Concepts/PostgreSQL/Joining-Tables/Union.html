Comparison of JOIN vs. UNION

JOINs combine tables side-by-side, producing wider rows that include columns from both sources. UNIONs stack result sets vertically, producing more rows by appending one result set under another.

Use Case: Archive Tables

It's common to split data into "current" and "archive" tables (for example current_employees and former_employees) to keep the active dataset small. A UNION lets you query both tables together to produce a unified list (for example, names and emails for all employees).

Rules for Using UNION

  1. Column Count: Each query in the union must return the exact same number of columns.
  2. Data Types: Columns in the same ordinal position must be compatible (you can't stack an integer on top of a string).
  3. Literals: Use literals to satisfy column counts or to add metadata (e.g., SELECT name, email, 'current' AS type).

Handling Duplicates (UNION vs. UNION ALL)

  • UNION (default): Removes duplicate rows, which incurs additional work (sorting/hashing) and a performance cost.
  • UNION ALL: Returns all rows including duplicates. It is faster because it skips duplicate elimination.

Performance tip: Prefer UNION ALL when you know duplicates are impossible from business logic or when you explicitly want to retain duplicates. Use UNION only when deduplication is required.

Example

SELECT name, email, 'current' AS type FROM current_employees
UNION ALL
SELECT name, email, 'former' AS type FROM former_employees;
